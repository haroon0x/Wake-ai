Wake · Pre-event dry run

Running Gemma offline on your phone.

Get Gemma 4 answering on an Android phone with no network, first with zero code to prove the hardware, then through the LiteRT-LM API to prove you can integrate it. Learning and assets only — none of this code goes into the hackathon repo.

The whole idea in one line

RAM is not the constraint; E2B is comfortable on 8 GB. The backend is, and it is phone-specific — so the choice of demo phone, not the code, is the real decision here.

The runtime

What LiteRT-LM actually is

Getting Gemma to run offline on the phone is the highest-risk unknown, so de-risk it before the event.

LiteRTGoogle's on-device ML runtime, the successor to TensorFlow Lite.
LiteRT-LMA layer on top, built only for LLMs: tokenization, KV-cache, chat sessions, streaming, GPU and NPU acceleration.
MediaPipe LLM APIThe older path, now maintenance-only. Google says new projects use LiteRT-LM, so target that.
Model format.litertlm files. Use litert-community/gemma-4-E2B-it-litert-lm from HuggingFace. E2B is smaller and faster, E4B is bigger and better.

Speed reality: E2B runs around 52 tokens/sec on the GPU (OpenCL) backend, but only 2 to 5 on CPU. GPU or NPU is what makes the demo feel instant. But the fast backends are not on every phone, which makes the choice of demo phone the biggest decision here.

Read before choosing a phone

Backend availability is the landmine

The GPU backend needs OpenCL, which many phones do not expose.

Pixel (Tensor G3/G4)No OpenCL. GPU backend silently fails, confirmed by open issues. Falls back to slow CPU.
Some Samsung/ExynosGPU init fails; hit or miss.
Snapdragon 8-seriesBest bet. Most likely to expose OpenCL, and the 8 Elite has an NPU reachable via the Qualcomm QNN delegate, which is even faster.

Backend priority to aim for: NPU via QNN on a Snapdragon 8 Elite, then GPU via OpenCL, then CPU via XNNPack as the streaming 2-to-5 tokens/sec floor.

Why this matters

CPU at 2 to 5 tokens/sec is not dead with streaming and short answers, but it is not the snappy demo. Know your phone's backend from the Stage 1 test, never discover it on stage. Plan the script around whatever backend your actual phone gives you.

Stage 0

Prep the phone, once

Ten minutes of setup that everything else depends on.

Check RAM and chipset

Settings then About phone. Want 8 GB or more. Snapdragon 8-series is the one you want; Pixel or Tensor means CPU only.

Enable developer options

Settings, About phone, tap Build number seven times.

Turn on USB debugging

Settings, Developer options, USB debugging on.

Install adb on the laptop

On WSL or Linux, apt install adb, plug in, run adb devices, accept the trust prompt on the phone.

Stage 1 · 15 minutes

Zero-code proof it runs

Confirm the hardware can run Gemma at usable speed before writing anything.

Install AI Edge Gallery

Google's official runner app, from the Play Store or the github.com/google-ai-edge/gallery releases.

Pick a Gemma model

Choose the LLM chat task, then the smallest Gemma, around E2B. Accept the HuggingFace Gemma license and download on home wifi, not at the venue.

Select GPU

Try to select the GPU backend. If it is greyed out or errors, your phone has no OpenCL and is running CPU. Note which backend actually ran.

Airplane mode, then chat

Confirm it answers with no network, note the tokens/sec, and check it does not overheat or crash after a minute.

Test all three phones

The fastest phone is your demo device. Record each one's chipset, backend, and tokens/sec.

Decision gate

If any phone gets GPU or NPU and feels snappy, commit to Android with that device. If the best phone is CPU-only, Android still works but keep LLM answers short and lean on the deterministic composer. If nothing is usable or all crash, fall back to the desktop plan.

Stage 2 · 1 to 2 hours

Prove you can drive the API

Learn the exact LiteRT-LM Android API so on the day you rewrite the integration fresh in about 30 minutes instead of debugging blind.

Get the model onto the phone

Download the .litertlm from litert-community/gemma-4-E2B-it-litert-lm, accept the license, then push it.

Build the official sample

Clone github.com/google-ai-edge/LiteRT-LM, open the Android sample in Android Studio, let Gradle sync so it pulls the dependency, and run over USB.

Read the lines that matter

Find model load, backend selection with a support check and CPU fallback, session creation, and the streaming generate call. That is the whole API you will reuse.

Modify one thing

Feed it a few fake captured events as context and ask "what was I just doing?" Watch it answer grounded in the context. Your hackathon AI path is proven.

# push the model where the sample expects it
adb shell mkdir -p /data/local/tmp/llm/
adb push gemma-4-E2B-it.litertlm /data/local/tmp/llm/
# match the exact filename/path in the sample README

Do not copy from memory

The LiteRT-LM API is young and class and method names shift between versions. The sample app on the tag you clone is the only trustworthy reference. Read it, not old blog posts, and take the Gradle coordinate from its build.gradle.

Gotchas

The traps to remember

GPU vs CPUIf speed is 2 to 5 tokens/sec you are on CPU. First check the backend is set to GPU, but if your phone has no OpenCL, CPU is all you get and there is no fix.
Cold startFirst load is slow as weights move into RAM. That is one-time per app start, not per message. Do not panic at cold start.
Venue wifiAssume unusable. Models, and the Gradle dependency cache from one full home build, must be local beforehand.
